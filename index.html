Lab Setup & Environment

Pre-Engagement // Est. 15–20 min // ZAP 2.15+ + DVWA on Docker

This lab targets  OWASP ZAP 2.15.0+  — UI paths reflect the current ZAP interface
Authorization required. All testing is performed against DVWA — a legal, intentionally vulnerable app running locally in Docker. Never use these techniques against systems you do not own or have documented written authorization for.
Pre-requisites checklist
  • OS: Kali Linux 2024.x / Parrot OS / Ubuntu 22+ (8 GB RAM minimum)
  • OWASP ZAP 2.15+ — sudo apt install zaproxy on Kali, or download from zaproxy.org
  • Docker Engine installed and running — sudo systemctl start docker
  • Firefox with FoxyProxy Standard extension installed
  • Port 80 free — stop any local web server: sudo systemctl stop apache2
Step 1 — Spin up DVWA in Docker
# Pull the DVWA image
$ sudo docker pull vulnerables/web-dvwa
Status: Downloaded newer image for vulnerables/web-dvwa:latest

# Run DVWA — maps container port 80 to host port 80
$ sudo docker run --rm -it -p 80:80 vulnerables/web-dvwa
Starting MySQL ... ok
Starting Apache httpd ... ok
DVWA running. Browse to http://127.0.0.1
Your container gets an auto-generated name (e.g. elastic_galois). Use it with docker stop, docker start, docker logs -f. Remove dead containers with docker rm <name>.
Step 2 — DVWA first-time database setup
  • Browse to http://127.0.0.1/setup.php
  • Click "Create / Reset Database" — wait for green Success messages
  • Login at http://127.0.0.1/login.php with admin / password
  • Click DVWA Security in the left menu → set to Low → click Submit
Dashboard loads with security level showing "Low" — setup complete.
Step 3 — Configure ZAP 2.15+ proxy & Firefox
ZAP 2.15+ reorganised the options menu. Use these exact paths.
  • Launch ZAP. When prompted about session persistence — choose "No, I do not want to persist this session" for now.
  • Confirm proxy: Tools → Options → Network → Local Servers/Proxies — should show 127.0.0.1:8080
  • Export CA cert: Tools → Options → Network → Server Certificates → click Save → save as zap-root-ca.cer
  • Import cert in Firefox: about:preferences#privacy → Certificates → View Certificates → Authorities → Import → select zap-root-ca.cer → tick "Trust this CA to identify websites"
  • In FoxyProxy: add proxy 127.0.0.1 port 8080 — enable it
  • Localhost proxy fix: Firefox about:config → search network.proxy.allow_hijacking_localhost → set to true
  • FoxyProxy: edit your ZAP entry → URL Patterns tab → add http://127.0.0.1/* as Whitelist pattern
Validation: browse to http://127.0.0.1/dvwa with FoxyProxy active — ZAP History tab should instantly show the GET request and the Sites tree should populate.

Phase 1 — Reconnaissance & Spidering

PTES Phase 2 // Est. 20 min // T1595 — Active Scanning

Concept: Why seed before spidering? 5 min

ZAP's spider follows links it discovers. If you give it a logged-in starting point it maps authenticated pages. If you let it start cold it gets stuck on the login redirect and discovers almost nothing. Manual seeding = handing ZAP a map before asking it to search the building.

  • Enable FoxyProxy → browse to http://127.0.0.1/dvwa — confirm ZAP History tab shows the request
  • Log in with admin / password
  • Click every left-menu item one by one: SQL Injection, XSS Reflected, XSS Stored, Command Injection, File Upload, Brute Force, CSRF, File Inclusion
  • On each page: submit at least one default input (hit Submit/Run without changing anything)
  • Watch ZAP Sites tree: http://127.0.0.1 should expand showing /dvwa/vulnerabilities/ with all sub-paths
Sites tree should show 8+ vulnerability paths. That's your seed. Now the spider has real links to follow.
Run the Standard Spider 8 minT1595.003

  • Start URL: http://127.0.0.1/dvwa/ — tick Recurse
  • Click Start Scan — watch the Spider tab at the bottom populate with URLs
  • When complete: note total URL count shown in the Spider tab footer

AJAX Spider — for JS-heavy apps 5 min

DVWA is mostly server-rendered so the standard spider is sufficient. Learn AJAX Spider now because every real-world React/Vue/Angular app needs it.

  • In ZAP menu bar: Tools → AJAX Spider
  • Start URL: http://127.0.0.1/dvwa/ — Browser: Firefox Headless
  • Click Start Scan — ZAP drives a real headless browser, executing JS and discovering dynamically-rendered links
  • Compare AJAX Spider results with standard Spider — note any additional URLs discovered
On real engagements: always run both spiders. Standard spider is fast and handles forms well. AJAX spider finds content behind JS-rendered navigation, SPAs, and lazy-loaded components.
🔵 Blue Team Overlay — Detection
  • Web logs: rapid sequential GET requests from single IP hitting every path — classic spider pattern
  • User-Agent: ZAP default UA contains "ZAP" — trivial WAF/proxy detection rule
  • SIEM rule: >50 unique URLs crawled from one IP in 60 seconds → alert
  • Mitigation: robots.txt rate limiting, honeypot URLs (canary links), WAF bot detection
  • NIST SP 800-53: SI-4 (System Monitoring), SC-5 (Denial of Service Protection)

Phase 2 — Active Scanning & Vulnerability Discovery

PTES Phase 4 // Est. 25 min // T1190 — Exploit Public-Facing Application

Active scanning fires real attack payloads at the target. This is why DVWA in Docker is the only acceptable target for this lab. On real engagements: always confirm active scanning is in scope and agreed in the RoE before proceeding.
Create a Custom Scan Policy 5 min

Default ZAP scan policy runs every check. For targeted web app testing, create a focused policy — faster scans, fewer false positives, more relevant results.

  • Analyse → Scan Policy Manager → click Add
  • Name it DVWA-WebApp-Lab
  • Enable categories: Injection, XSS, Path Traversal, Remote File Inclusion, Server Side Include
  • Set threshold to Medium and strength to High for Injection checks
  • Disable: Buffer Overflow, Format String (not applicable to PHP web apps)
  • Click OK to save the policy
Saving custom scan policies = professional practice. On real engagements you build one per tech stack (PHP, Java, Node.js) and reuse across engagements.
Run Active Scan Against DVWA 15 minT1190OWASP A03/A05
  • In ZAP Sites tree: right-click http://127.0.0.1/dvwa → Attack → Active Scan
  • In the Active Scan dialog: select your DVWA-WebApp-Lab policy from the Policy dropdown
  • Tick Recurse — this scans all discovered child URLs, not just the entry point
  • Click Start Scan — estimated runtime: 10–15 minutes
  • Watch the Active Scan tab progress bar and the Alerts tab building up in real time

Read an Alert Properly 5 min

Every ZAP alert contains the evidence you need to write a finding. Learn to read it properly now.

  • Click any High alert in the Alerts tab
  • Read the Alert pane at the bottom — note: URL, Parameter, Attack, Evidence, Description, Solution
  • Click the Request tab in the bottom pane — see the exact HTTP request ZAP sent including the payload
  • Click the Response tab — see the server response that triggered the alert
  • Right-click the alert → Open in Browser — ZAP replays the attack in your browser
The Request + Response tabs ARE your evidence. Screenshot them for every High/Critical finding before writing your report.
🔵 Blue Team Overlay — Detection
  • Active scan produces thousands of 4xx/5xx errors — extremely visible in web access logs
  • WAF: SQL Injection patterns (UNION, SELECT, quotes, comment sequences) in request parameters
  • WAF: XSS patterns — script tags, javascript: URIs, event handler attributes in input fields
  • SIEM: Alert on >200 4xx responses from single IP in 60 seconds
  • EDR: Unusual child processes spawned from web server (e.g. cmd.exe, /bin/sh as child of httpd)
  • Mitigation: WAF with OWASP CRS ruleset, rate limiting (100 req/min/IP), IPS signatures
  • NIST SP 800-53: SI-10 (Information Input Validation), SC-5, RA-5 (Vulnerability Scanning)

Phase 3 — Manual SQL Injection

PTES Phase 5 // Est. 20 min // T1190 | OWASP A03:2021 — Injection

Concept: What is SQL Injection?

SQL Injection occurs when user-supplied input is concatenated directly into a SQL query without sanitization. The attacker injects SQL syntax that changes the query logic — extracting data, bypassing authentication, or in some cases executing OS commands.

# Vulnerable PHP code — DO NOT write code like this
$query = "SELECT * FROM users WHERE id = '" . $_GET['id'] . "';";

# Attacker input: 1' OR '1'='1
# Resulting query:
SELECT * FROM users WHERE id = '1' OR '1'='1';
-- Returns ALL users because '1'='1' is always true
Exercise A — Basic SQLi with ZAP Requester 10 minT1190
  • Browse to DVWA → SQL Injection → enter 1 in the User ID field → click Submit
  • In ZAP History tab: find this GET request → right-click → Send to Requester
  • In the Requester tab: locate id=1 in the URL — change the value to 1' OR '1'='1
  • Click Send — inspect the Response tab. Does it return more than one user?
  • Now try: 1' UNION SELECT user,password FROM users-- -
  • Screenshot the response showing user hashes — this is your critical evidence
# Payload 1 — Authentication bypass / data dump
URL param: id=1' OR '1'='1
Expected: All users returned from the database

# Payload 2 — UNION-based data extraction
URL param: id=1' UNION SELECT user,password FROM users-- -
First Name: admin
Surname: 5f4dcc3b5aa765d61d8327deb882cf99
# That hash is MD5("password") — confirmed credential extraction
Exercise B — SQLi Fuzzing with ZAP Fuzzer 10 min
  • In ZAP History: right-click the SQLi GET request → Attack → Fuzz
  • In the Fuzzer dialog: highlight the value 1 in the id=1 parameter → click Add
  • Payload type: File Fuzzers → expand jbrofuzz → select SQL Injection
  • Click Add → OK → Start Fuzzer
  • In Fuzzer results: sort by Size Resp. Body — payloads returning different response sizes = potential injection points
  • Click any interesting result → check Response tab for evidence of data leakage or errors
Different response sizes = the SQL query behaved differently = injection is occurring. This is the fundamental signal in blind SQLi detection.

🔵 Blue Team Overlay — Detection
  • WAF: SQL keywords in GET/POST params — UNION, SELECT, INSERT, DROP, OR 1=1, comment sequences (-- -, #, /*)
  • Database logs: unusual queries with tautologies or UNION statements
  • App logs: unexpected query errors (MySQL error messages in response body = SQLi AND information disclosure)
  • SIEM: Correlate 4xx spikes with DB error log entries from same timeframe
  • Mitigation: Parameterized queries (PDO/MySQLi), WAF CRS rule 942xxx series, least-privilege DB accounts
  • NIST SP 800-53: SI-10 (Input Validation), AC-3 (Access Enforcement)

Phase 4 — Cross-Site Scripting (XSS)

PTES Phase 5 // Est. 15 min // T1059.007 | OWASP A03:2021

Concept: Reflected vs Stored XSS

Reflected XSS: Malicious script in the URL/request — reflected back in the response and executed in the victim's browser when they click a crafted link. Impact: session hijacking, credential theft, phishing overlay.

Stored XSS: Malicious script saved in the database — executes for every user who views the infected page without needing to click anything. Higher severity: affects all users persistently.

Exercise A — Reflected XSS 8 minT1059.007
  • Browse to DVWA → XSS (Reflected)
  • Enter <script>alert('XSS-Reflected')</script> in the Name field → click Submit
  • Alert box appears? Reflected XSS confirmed — screenshot it
  • In ZAP History: find the request → right-click → Send to Requester
  • In Requester: modify the name= parameter to: <img src=x onerror=alert(document.cookie)>
  • Click Send → check Response — does it show your session cookie in the alert?
  • Try bypass payload: <svg/onload=alert(1)> — works if script tags are filtered but SVG is not
# XSS payload escalation chain — test each in order
Payload 1: <script>alert(1)</script>
Tests: basic script injection, no filtering

Payload 2: <img src=x onerror=alert(document.cookie)>
Tests: img event handler — bypasses basic script tag filters

Payload 3: <svg/onload=alert(document.domain)>
Tests: SVG vector — bypasses HTML sanitizers that miss SVG namespace

If any payload executes — XSS is confirmed. Document which payload succeeded.
Exercise B — Stored XSS 7 minOWASP A03
  • Browse to DVWA → XSS (Stored)
  • In the Message field enter: <script>alert('Stored-XSS')</script> — Name field: anything
  • Click Sign Guestbook — alert fires immediately (it's stored and reflected back to you)
  • Refresh the page — alert fires again without resubmitting. This is stored XSS.
  • Open an incognito window (no FoxyProxy) → browse to http://127.0.0.1/dvwa/vulnerabilities/xss_s/ → login → observe alert fires for a fresh session
Stored XSS is a Critical finding. Every user who visits the page executes the attacker's code. In a real app this could steal every active session token.
In ZAP: use Tools → Options → Active Scan → Max alerts per rule to control how many XSS instances ZAP reports per page. Default is 10 — on large apps raise this to find all injection points.

Phase 5 — Command Injection & Remote Code Execution

PTES Phase 5–6 // Est. 15 min // T1059 | OWASP A03:2021

Concept: OS Command Injection

Command injection occurs when user input is passed directly to an OS shell function without sanitization. If a PHP app calls shell_exec("ping " . $_GET['ip']), an attacker appends a shell separator (;, &&, |) to inject additional commands. This is Remote Code Execution — the highest severity web finding.

Exercise — Command Injection via ZAP Requester 15 minT1059
  • Browse to DVWA → Command Injection → enter 127.0.0.1 → click Submit → confirm ping output appears
  • In ZAP History: find this POST request → right-click → Send to Requester
  • In Requester: locate ip=127.0.0.1 in the request body → change to 127.0.0.1; id → Send
  • Check Response tab — do you see uid=33(www-data)? That is confirmed RCE.
  • Try: 127.0.0.1 && whoami (Windows-style separator — tests for platform)
  • Try: 127.0.0.1 | cat /etc/passwd — extracts system user list
  • Screenshot every successful output — each is a distinct piece of evidence
# Command injection payload chain — test each separator
Payload 1 (semicolon): 127.0.0.1; id
uid=33(www-data) gid=33(www-data) groups=33(www-data)

Payload 2 (AND): 127.0.0.1 && cat /etc/passwd
root:x:0:0:root:/root:/bin/bash
daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
... (system users listed)

Payload 3 (pipe): 127.0.0.1 | uname -a
Linux dvwa 5.15.0-91-generic #101-Ubuntu ... (kernel fingerprint)

# RCE confirmed. This is your highest severity finding.
In this lab: stop at information gathering. Do NOT attempt to establish a reverse shell or escalate further — not in scope for this beginner lab. Document the RCE evidence and write the finding.
Bonus: Fuzz Command Injection Separators 5 min
  • ZAP History: right-click the command injection POST → Attack → Fuzz
  • Highlight the value 127.0.0.1 → Add → File Fuzzers → jbrofuzz → OS Command Injection
  • Start Fuzzer → sort results by response body size — larger responses = command executed
  • Review the payloads that returned unusual response sizes — these are your additional injection vectors

🔵 Blue Team Overlay — Detection
  • Sysmon Event ID 1: unusual child process spawned by apache2/php — e.g. /bin/sh, id, cat, whoami as child of httpd
  • auditd: execve() syscall calls by www-data process spawning unexpected binaries
  • EDR: behavioural rule — web server process spawns shell interpreter → Critical alert
  • Network: unexpected outbound connections from web server IP (reverse shell beacon pattern)
  • Mitigation: AppArmor/SELinux profile for Apache, disable shell functions in php.ini (disable_functions = exec,shell_exec,system,passthru,popen)
  • NIST SP 800-53: CM-7 (Least Functionality), SI-10, AC-6 (Least Privilege)

ZAP Automation — Level Up Optional

ZAP 2.15+ includes the Automation Framework — run full scan + report generation from a YAML config file. Essential for CI/CD pipeline integration.

# ZAP Automation Framework — minimal config (automation.yaml)
env:
  contexts:
  - name: dvwa-context
    urls: ["http://127.0.0.1/dvwa/"]
jobs:
  - type: spider
    parameters: {maxDuration: 2}
  - type: activeScan
    parameters: {maxDuration: 10}
  - type: report
    parameters: {template: traditional-html, reportFile: report.html}

# Run it
$ zap.sh -cmd -autorun automation.yaml
